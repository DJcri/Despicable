# AGENT.md

## Companion file note

- `AGENT.md` is the detailed policy document.
- `AGENTS.md` is the shorter quick-reference summary for fast review and tooling that prefers the plural filename.
- Keep both files aligned when updating guardrails.

This repository favors **stability, consistency, and additive changes** over clever rewrites.

## Primary operating rules



- If you change the **UI Framework**, update the UI docs (`Docs/UIFramework_RulesOfTheRoad.md` and `Docs/UIFramework_Cookbook.md`) in the same pass.
- If a UI pattern repeats (or is likely to), prefer adding a small **UIFramework helper/blueprint** (additive, measure-safe) rather than cloning code.
- Prefer the **smallest correct change** that fits the existing architecture.
- Prefer **extension, composition, wrappers, and focused new files** over broad refactors.
- Do not rename, move, or reshape stable APIs unless there is a clear payoff and a compatibility bridge.
- Do not create a second source of truth for settings, runtime state, or cached data.
- Every piece of mutable state must have **one obvious owner**.
- New code must follow the repository's established naming, folder, and lifecycle patterns.

## Localization rules (must-follow)

Localization is enforced by a build-time audit.

- Any new **player-facing** text in C# (UI labels, buttons, tooltips, messages, dialog text) must use `Translate()`:
  - Use `"Key".Translate(...)` and add the key to `Languages/English/Keyed/*.xml`.
  - Prefer stable semantic keys when practical; updating the English text should not require renaming keys.
- Def text changes (`label`, `description`, `reportString`, etc.) must keep **English DefInjected mirrors** in sync:
  - The build audit auto-generates missing entries into `Languages/English/DefInjected/<DefType>/AutoGenerated_DefInjected.xml`.
- Never move Core-only keys into NSFW (Core must not depend on NSFW language files).
- Core must not reference NSFW interaction icon paths (e.g. `UI/Interaction/...`) or ship those textures; they belong in the NSFW add-on (use `TexIconsNSFW` there if needed).
- Treat `Languages/English/Keyed/CodeStrings_UI.xml` as a normal Keyed file (not a scratch TODO).

- Avoid `->` navigation arrows and raw `<` / `>` characters in translation strings (they can trigger RimWorld translation load errors). Use `→` or `/` instead.
- Guardrails also validate: duplicate keys and malformed `{0}` placeholders/braces.
Build-time audit:
- Script: `Tools/LocalizationAudit.ps1`
- Skip switch (temporary only): `D2SkipLocalizationAudit=true`

## Refactor restraint rules

Default to **not refactoring unrelated code** while implementing a feature or fix.

Refactors are allowed only when at least one of these is true:

- the current code blocks the requested work
- the current code causes a bug, correctness issue, or repeat breakage
- the change removes a duplicate source of truth or unsafe mutable global state
- the file has crossed a clear readability threshold and can be split at an obvious seam
- a wrapper can preserve compatibility while the internals are cleaned up

When refactoring is justified:

- keep the blast radius narrow
- preserve behavior unless the task explicitly changes behavior
- preserve public method names where practical, or keep compatibility wrappers
- prefer introducing a new owner/helper/file rather than rewriting a stable one in place
- stop once the problem is solved; do not turn a local repair into a roaming rewrite

## State ownership rules

Before adding any field, classify it into exactly one bucket:

1. **Constant / default**
   - Use `const` or `static readonly`.
   - Do not use mutable `static` for values that are effectively defaults.

2. **Settings / config**
   - Store in settings/config.
   - Read through a single accessor.
   - Do not mirror settings into mutable runtime globals.

3. **Per-instance runtime state**
   - Keep on the owning component/class.
   - Prefer this over shared state.

4. **Shared ephemeral runtime state**
   - Allowed only when there is a clear cross-instance need.
   - Must live in a named owner (`*RuntimeState`, registry, cache owner, etc.).
   - Must define `ResetRuntimeState()`.

5. **Cache**
   - Name what it stores.
   - State the owner.
   - Define clear invalidation/reset behavior.

## Static field rules

Avoid new mutable `static` fields by default.

Mutable `static` is allowed only for:

- immutable-once-initialized startup detection guarded by one owner
- reflection caches and lookup caches
- warn-once registries
- thread-local guards / scoped render context
- explicit shared runtime-state owners with `ResetRuntimeState()`

Mutable `static` is **not** acceptable for:

- copied settings values
- convenience "current object" globals without tight scoping
- duplicate runtime flags that shadow another owner
- vague catch-all state blobs

If a mutable `static` is introduced, document in one sentence:

- who owns it
- why it must be shared
- how it resets

## Consistency rules

- Booleans start with `Is`, `Has`, `Can`, or `Should`.
- Methods are verb-first.
- Runtime lifecycle reset methods are named `ResetRuntimeState()`.
- Cache/registry names must describe key/value shape, not use vague names like `data`, `stuff`, `temp`, or `misc`.
- Prefer explicit, descriptive names over short clever ones.
- Keep the same concept named the same way across files and subsystems.

## File placement rules

Place new code in the most specific folder possible.

- `Runtime/` for shared ephemeral runtime state, registries, and reset hubs
- `Patches/` for Harmony/patch files
- `Compat/` or `Integrations/` for external mod or framework integration
- `UI/` or UI-specific folders for UI code
- `Dev/` for diagnostics, debug tooling, and developer-only helpers
- `Comp/` for component-owned behavior/state

Do not keep growing a generic utility blob if a more specific home exists.

## Growth thresholds

Pause and do a small structural cleanup when any of these happens:

- a file grows past roughly 300 to 400 lines and has mixed responsibilities
- a method grows past roughly 40 lines and becomes hard to scan
- a class gains a second unrelated responsibility
- you are about to add a second or third shared dictionary/set for one feature
- you are copying logic from another subsystem instead of reusing or extracting a seam

The preferred fix is a **small, local split at an obvious seam**, not a sweeping rewrite.

## Implementation workflow

For new features:

1. choose the correct folder
2. choose the state owner
3. add settings/config access if needed
4. add `ResetRuntimeState()` if introducing shared ephemeral state
5. then implement behavior

For changes to existing features:

1. match the existing naming dialect
2. use existing owners and helpers first
3. add wrappers rather than breaking stable callers
4. keep changes local unless a broader change is required

## Before finishing a change

Run the repository guardrail checker and quickly confirm:

- Did I add any new mutable `static` fields?
- Does every new shared cache or registry define `ResetRuntimeState()`?
- Did I create a duplicate source of truth?
- Did I add to a blob file when a focused file would be cleaner?
- Did I change stable APIs more than necessary?
- Did I follow the existing naming and folder conventions?

The best change is the one that solves the problem cleanly **without making future cleanup necessary**.

## Localization guardrails
- Keyed rule: every Translate() key must exist in Languages/English/Keyed/*.xml.
- Unused Keyed keys are pruned unless kept in Tools/LocalizationKeepKeys.txt.
- DefInjected mirrors are auto-generated for English.

- DefInjected orphan rule: DefInjected nodes whose defName no longer exists in Defs are pruned from AutoGenerated_DefInjected.xml and must not exist in other DefInjected files unless whitelisted in Tools/DefInjectedKeep.txt.
